# Real-engine evaluation harness

This directory contains the harness for §Live Evaluation in the
paper. The script provisions a PostgreSQL 17 container, runs
pgbench (TPC-B-like, scale-10), injects a single *known structural
fault* (DROP INDEX on `pgbench_accounts_pkey` at `t = 30 s`), and
captures a SHA-256-pinned residual tape per replication. It then
runs the `replay_tape_baselines` binary to score DSFB and three
published change-point baselines (ADWIN, BOCPD, PELT) on the same
tape under the same ground-truth window, using the *same*
`evaluate()` path that grades the offline bake-off in §7.

## Invocation

```
cd crates/dsfb-database
cargo build --release --features "cli report live-postgres"
bash experiments/real_pg_eval/run.sh
```

Total wall time ≈ 12 minutes for `N_REPS=10` replications of 70 s
each plus init + aggregation.

Requires: `podman` (see `provenance.txt` for the pinned image
digest) and a `pgbench` binary on `$PATH` (PG client-tools ≥ 14).

## Outputs

Under `experiments/real_pg_eval/out/`:

```
out/
├── provenance.txt           # container digest, pgbench version, crate SHA
├── r01/                     # replication 1
│   ├── live.tape.jsonl
│   ├── live.tape.jsonl.hash
│   ├── live/                # per-replication episodes CSV + poll log
│   ├── bakeoff.csv          # one row per (detector, motif); header pins tape + GT SHAs
│   └── *.log
├── r02/ ... r10/            # replications 2–10
└── summary.csv              # mean / stddev / 95% CI bootstrap(B=1000), per (detector, motif, metric)
```

The summary CSV has exactly `4 detectors × 5 motifs = 20 rows`.

## Reproducibility discipline

* Container image is pinned by digest (SHA-256) in `run.sh`.
* `pgbench` version is captured to `provenance.txt`.
* Crate git SHA is captured.
* Every captured tape is SHA-256-pinned by its sidecar `.hash`
  manifest; `replay_tape_baselines` verifies the hash before
  scoring and refuses to proceed on mismatch.
* The n=10 *summary* (mean / CI) is the reproducible artefact —
  individual tape bytes are not reproducible across re-runs
  (paper §10 non-claim #7). Two tapes captured under the same
  workload + same fault will differ in sampling jitter and
  counter-advance timing.

## Ground truth

`ground_truth.json` pins:

* `tape_sha256`: expected SHA-256 of the tape this ground-truth
  annotation was authored against. The replay refuses to score a
  tape whose hash does not match.
* `fault_description`: human-readable fault label (DROP INDEX).
* `windows[]`: one fault window per motif, with the exact
  residual-channel string the fault was planted on. For
  pg_stat_statements the channel is `md5(queryid::text)` — the
  adapter's anonymised qid.

## Pinned fixture tapes

The paper's figures (`live_real_pg_trajectory.png`,
`live_determinism_overlay.png`) are byte-deterministic functions
of two *pinned* captured tapes that live under

```
paper/fixtures/live_pg_real/replication_01.tape.jsonl
paper/fixtures/live_pg_real/replication_02.tape.jsonl
```

These two tapes are **not** regenerated by `run.sh`; they are
captured artefacts. The n=10 summary table cited in §Live
Evaluation **is** regenerated (up to CI overlap) by `run.sh`.

## What this harness does *not* do

* It does not cover multiple fault classes (only DROP INDEX).
  A fault-taxonomy sweep is a separate pass (see §Limitations).
* It does not run in CI. CI runs the pinned-tape reproducibility
  test (`tests/live_replay_baselines_reproducibility.rs`) and the
  feature-gated live-postgres unit tests, neither of which need a
  running engine.
* It does not validate against a production workload. The
  within-container variance captured here is a within-container
  bound, not a production bound.
